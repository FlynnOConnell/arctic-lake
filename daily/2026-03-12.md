## Workers lifecycle

MATLAB (multiFuse_tiled.m / multiFuse.m)
Parallelization: Sequential loop over timepoints/tiles. External job scheduler handles parallelism (not parfor). Each MATLAB worker is a completely separate process with its own memory space.

Per-worker peak memory: 3 full volumes + small 2D masks

The lifecycle within one worker (one timepoint, one camera pair):

Load primaryDataArray{1} + primaryDataArray{2} (2 volumes)
Optionally load 2D sliceArray masks (tiny)
In-place intensity correction on the same arrays (no copies)
Create fusedStack (1 more volume) — peak is here: 3 volumes
Pixel-by-pixel blending loop writes directly into fusedStack
clear fusedStack after write to disk
Camera pairs are processed sequentially within a worker — after finishing pair 1, those arrays go out of scope before pair 2 starts
Key efficiency: MATLAB corrects volumes in-place (primaryDataArray{1,2} are modified, not copied) and the blending loop writes directly into the pre-allocated fusedStack. No intermediate copies are created.

Python (_fuse_camera_pair in fusion.py)
Parallelization: ProcessPoolExecutor(max_workers=config.workers). All timepoints are submitted at once to the pool (line 483-489). Each worker runs _fusion_tp_worker which processes camera pairs sequentially.

Per-worker peak memory: ~8-10 full volumes + masks

Here's why — tracing through _fuse_camera_pair:

Line	Variable	Size	Notes
804-806	vol0, vol1	2 volumes	loaded from disk
812	vol1 flip	+1 copy	np.flip(...).copy()
815-818	vol1 rotate/flip	+copies	each transform allocates
883	vol1_transformed	+1 volume	apply_camera_transform creates new array
884	mask1_transformed	+1 float32 mask	mask transform
888-889	vol0_sub, vol1_sub	+2 volumes	subtract_background creates new arrays
895	vol0_sub, vol1_corrected	potentially +1	apply_intensity_correction
905-906	slice_mask0, slice_mask1	+2 2D masks	small
915	avg_mask	+1 2D mask	small
980-988	fused	+1 volume	blend_views output
1009-1011	projections	+3 2D arrays	small
1054	combined_mask	+1 bool volume	3D mask
1056	write_volume(combined_mask)	+pyramid copies	pyramid generation
The critical difference: Python variables are never explicitly freed. vol0, vol1, vol1_transformed, vol0_sub, vol1_sub, vol1_corrected, mask0, mask1, mask1_transformed, and fused all coexist in memory simultaneously because they're all referenced in the same function scope until _fuse_camera_pair returns.

Side-by-Side Summary
Aspect	MATLAB	Python
Peak per worker	3 volumes	8-10 volumes
In-place ops	Yes (modifies loaded arrays)	No (creates new arrays at each step)
Task submission	One at a time (sequential loop)	All at once (pool.submit in loop)
Variable cleanup	clear statements mid-function	Only on function return
Pyramid generation	Not done during fusion	Done in same worker (write_volume with pyramid=True)
3D mask write	Not saved	combined_mask = extra bool volume
