# Cellpose segmentation pipeline on isoview fused volumes

## dataset
- **Source**: `E:\datasets\isoview\2026-01-23_drosophila\Dme_E1_57C10_GCaMP6s_Simultaneous_6p22_20150528_163012.
corrected_zarr\Results\MultiFused_auto\SPM00\TM######\SPM00_TM######_CM00_CM01_VW00.zarr`
- **61 timepoints** (TM000000 – TM000060)
- **Per-volume shape (raw fused)**: `(38, 1848, 768)` uint16

## 1. Foreground mask tuning
Built an `ipywidgets` slider over isoview's `segment_foreground` to find the
threshold that cleanly separates signal from background. Reused isoview's
internal `_imgaussian_anisotropy` precomputation so the slider only re-applied
the level cutoff (sub-second updates).

- **Settled on `threshold = 0.6`** (isoview's adaptive 0–1 scale)
- Other params: `kernel_sigma=5.0`, `kernel_size=11`, `mask_percentile=1.0`,
`scaling=0.40625/(6.5/16)=1.0`, `subsample_factor=100`
- Resulting foreground fraction on TM000000: ~24%

## 2. Crop to ROI
Bounded the volume to the actual brain region using ImageJ-derived coordinates.
After resolving an axis-order confusion (numpy ZYX vs ImageJ display orientation),
the final crop:

- `y_range = (350, 1600)`  → 1250 px
- `x_range = (150, 650)`   → 500 px
- `z_range = (8, 36)`      → 28 slices
- **Cropped shape: `(28, 1250, 500)`** = ~17.5 M voxels per timepoint

## 3. Per-timepoint preprocessing pipeline
Single function (`process_and_segment`) per timepoint, called in a loop over all
61 timepoints. Each run:

1. Load fused `SPM00_TM#####_CM00_CM01_VW00.zarr`
2. Apply `segment_foreground(threshold=0.6)`
3. Zero out background (`np.where(mask, vol, 0)`)
4. Crop to (28, 1250, 500)
5. Run **cellpose-SAM 2D per-plane** (initial pass)
6. Save as a zarr group containing `volume` and `labels_2d`

Output organized as `{tm}.zarr` per timepoint, each a zarr group with `volume`
+ `labels_2d` arrays. Stored in `E:\datasets\isoview\2026-01-23_drosophila\cellpose\`.

## 4. Cellpose method comparison

### probe runs on 3 timepoints (TM000000, TM000010, TM000020)
Compared two strategies head-to-head and saved to `../data/probe/`:
- **`stitch_threshold=0.5`**: 2D per-plane + IoU linking across z
- **`do_3D=True`** with `anisotropy=1.0`, `flow3D_smooth=2`

`do_3D` produced visually superior segmentation (smoother boundaries, fewer
oversplit cells across z) → chose to commit to the 3D path.

### initial full do_3D run (slow)
- Settings: `do_3D=True`, `diameter=10`, `batch_size=8`, `tile_overlap=0.1`,
`flow3D_smooth=2`, `cellprob_threshold=-1.0`, `normalize={"norm3D": True, "percentile": [1, 99]}`
- **~50 min/timepoint**, ETA 42 hours for all 61
- Completed TM000000–TM000010 (11 timepoints, 95–146 cells/volume) before stopping

### speed optimization attempts (failed)
Tried `batch_size=32`, `tile_overlap=0.05`, `flow3D_smooth=0` combinations.
**Failed with `Unable to allocate 108 GiB`** for the resampled flow tensor of
shape `(1286, 3750, 1500, 4)` — root cause was Windows commit-charge ceiling
(same WinError 1455 family that hit suite3d earlier), not actual physical RAM.
The pagefile would need bumping to ~64 GB to make the original do_3D path
sustainable at higher batch sizes.

### diameter-sweep benchmark (do_3D, fits in current commit budget)
Clean-subprocess benchmark on TM000000:

| config | time | cells | recall vs d10 baseline |
|---|---|---|---|
| `d10 b8` (baseline)  | ~50 min  | 136 | 100% |
| `d15 b16`            | 18.6 min | 122 | 90%  |
| `d20 b16`            | 10.5 min | 83  | 61%  |

`d15 b16` is the recommended speed/accuracy sweet spot — **2.7× faster than
baseline** with 90% cell recall. `d20` is a quality cliff. Diameter is the
primary lever because cellpose internally rescales by `30/diameter`, and a
smaller rescale factor means proportionally less compute and less memory.

### stitch path (fast fallback)
Permissive stitch run on TM000000, written into the same zarr group as the
existing `labels_3d`:

| config | time | cells | array name |
|---|---|---|---|
| `stitch_threshold=0.4`, `cellprob=-1.5`, `min_size=20` | 2.7 min | 202  | (probe dir) |
| `stitch_threshold=0.4`, `cellprob=-6`, `flow=0`, `min_size=5` | 2.7 min | 1328 | `labels_permissive` |

The "permissive" run intentionally trades precision for recall to surface
anything cellpose's network has any opinion on — useful as a noise-floor
reference, not as a final segmentation.

## 5. Output zarr structure
Each `{tm}.zarr` group on E: now contains some subset of these arrays
(per-timepoint, depending on which runs have completed):

- `volume`            — uint16, the masked + cropped fused volume
- `labels_2d`         — int32, original per-z-plane cellpose 2D labels
- `labels_3d`         — int32, do_3D=True (TM000000–TM000010 only)
- `labels_permissive` — int32, permissive stitch (TM000000 only)

## 6. Visualization in napari
Wrote a `LazyStack` wrapper class that exposes a list of 3D zarr arrays as a
single 4D array-like object via `__getitem__` forwarding. Napari pulls one
`(t, z)` slice at a time, so RAM stays flat regardless of timepoint count
(no dask required). Used with both image and label layers, sharing T and Z
sliders.

For 3D rendering, set `scale=(0.40625, 0.40625, 0.40625)` on every layer
(image and labels must match) and `viewer.dims.ndisplay = 3`. The volume
displays as a thin physical slab (~11.4 × 508 × 203 µm) — that's the actual
physical geometry, not a display bug.
