## Main Projects

### mbo_utilities

- Parallel processing improvements and testing, SLURM batch scripts, better logging

  - Pick # cores based on filling GPU memory (4 zplanes ~ fill a single L40 GPU)
  - I/O is fully parallel (tiff -> suite2p binary) -> Pass 4 zplanes to L40 node, 4 zplanes to anothe L40, remaining 2 zplanes to last GPU

### LBM-Suite2p-Python

### IsoView

- v0.2.0: performance improvements, several bugfixes
- Isoview-python exceeds MATLAB performance on Tiled midgut dataset (tiff vs zarr v3 level 3 compression + 5 pyramids)
  - correct_stack: 18.4 min (mat) vs 12.5 min (py): 5 workers
  - multi_fuse: 13.8 min (mat) vs 10.2 min (py): 5 workers
  - MATLAB caps around 60% memory usage
  - Python is near 90% no matter what
- MATLAB unit of work is 1 timepoint, Python is 1 camera (correct_stack) or 1 view (multi_fuse)
  - Reason: 1 timepoint / 1 tile limits the number of cpu's that we can take advantage of
  - MATLAB, in correct_stack, parallelizes over Z where we vectorize instead
  - Need to add per-core memory estimation to Isoview

## Compute/Storage Server/HPC

- scp from hpc -> RBO-W2: `~240-270MB/s`
- repositories in scratch/mbo_soft/repos
  - mbo_utilities, mbo_server_configs, lbm_suite2p_python
- environments in scratch/mbo_soft/evs
  - mbo: mbo_utilities, lbm_suite2p_python, isoview, cellpose, suite2p, torch+cu126

- node: all 16 workers (4 GPUs Г— 4 planes, 2 wasted workers) live on one machine. So that machine's disk is reading 16 planes' worth of raw data at once, and its 48 CPUs are split 16 ways
- array: those 4 GPUs sit on 4 separate machines, 4 workers each. So each machine's disk reads only 4 planes, and its CPUs are split only 4 ways
- I/O is the bottleneck here, much slower when competing for disk bandwidth
